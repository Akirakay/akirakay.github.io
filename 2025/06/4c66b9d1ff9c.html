行为表现

REQUIRED / REQUIRED

REQUIRED: 如果当前有事务，就加入该事务；否则新建一个事务。
内外层事务传播行为均为 REQUIRED ，此时所有的事务方法处于一个事务之中，要么同时提交，要么同时回滚

分析

方法 placeOrder 执行过程：

  1. placeOrder 执行时开启事务A，执行新建订单，未提交
  2. 调用 deductStock 减库存，合并 deductStock 事务到事务A中
  3. deductStock 中，执行修改库存，手动抛出异常，事务回滚
  4. placeOrder 回滚新建订单数据

事务的执行链路：

如果 deductStock 正常执行，则会提交事务；如果 deductStock 抛出异常：
1） deductStock 声明了事务，方法内的数据库操作会回滚
2）placeOrder 和 deductStock 同属一个事务，placeOrder 也会执行回滚，保证了事务的原子性

如果 deductStock 异常被 catch 了，此时事务是如何处理的：

根据事务的原子性， deductStock 方法的异常触发回滚操作会引起整个事务的回滚。

因此，Spring 在处理事务过程中，当事务的传播性设置为REQUIRED，在整个事务的调用链上，任何一个环节抛出的异常都会导致全局回滚。

REQUIRED / REQUIRES_NEW

REQUIRES_NEW: 无论是否存在事务，总是新建一个事务，原来的事务会被挂起。

分析

方法 placeOrder 执行过程：

  1. placeOrder 执行时开启事务A，执行新建订单，未提交
  2. 调用 deductStock 减库存， deductStock 传播机制为：REQUIRES_NEW，因此挂起事务A，开启新事务B
  3. deductStock 中，执行修改库存，手动抛出异常，事务B回滚
  4. placeOrder 接收到 deductStock 传递的异常，事务A回滚新建订单数据

如果 placeOrder 中吞掉异常，会是什么结果：

SUPPORTS

SUPPORTS : 如果当前有事务，就加入事务；否则就以非事务方式执行。
外层有事务，内层为 SUPPORTS

分析

placeOrder 开启事务，SUPPORTS 为支持事务，所以 deductStock 在事务中运行，行为类似于 REQUIRED

如果 placeOrder 不在事务中，操作又是什么结果：

外层无事务，则内层也以非事务形式执行。
如果异常被吞掉，则订单状态也会被修改为：CONFIRMED

REQUIRED / NOT_SUPPORTED

NOT_SUPPORTED: 不支持事务，存在事务则挂起

REQUIRED / NEVER

NEVER: 不支持事务，如果外层开启事务，调用内层方法会抛出异常，内层方法内逻辑不会执行
具体执行结果根据外层是否吞掉异常来决定事务的回滚与提交

REQUIRED / MANDATORY

MANDATORY: 必须在事务中执行：

  • 如果外层方法没有开启事务，内层方法抛出异常
  • 如果外层方法开启事务，内层方法加入事务，在同一事务中执行

REQUIRED / NESTED

NESTED: 嵌套事务：

  • 如果外层方法不存在事务，内层方法的规则与REQUIRED 一致。
  • 如果外层方法存在事务，内层方法作为外层方法事务A的子事务执行，两个方法是一起提交，但子事务是独立回滚。
    内层方法抛出异常，则会回滚方法的所有操作，但不影响外层事务方法。（外层方法需要try-catch子事务，避免异常传递到父层事务）
    外层方法回滚，则内层方法也会回滚。
  • 该传播性的特点是可以保存状态点，当回滚时，只会回滚到某一个状态点，保证了子事务之间的独立性，避免嵌套事务的全局回滚。

总结

传播行为 是否创建新事务 若有现成事务 用途
REQUIRED 是（如果没有） 加入现有事务 默认，常用
REQUIRES_NEW 是 挂起当前事务 独立操作，互不影响
NESTED 否 嵌套（保存点） 局部失败处理
SUPPORTS 否 加入现有事务 读操作或非关键性操作
NOT_SUPPORTED 否 挂起当前事务 导出、耗时任务等
NEVER 否 抛异常 严格非事务要求
MANDATORY 否 加入现有事务 必须由外层开启事务
